1. Overview

The Wazuh Claude AI Alert Analyzer is a Python-based security automation tool that monitors Wazuh alerts in real time, sends high-severity and custom-rule alerts to the Anthropic Claude AI API for intelligent analysis, and writes structured investigation reports directly into the Wazuh OpenSearch index — making them visible and searchable in the Wazuh Dashboard.

What it does

  • Reads alerts from /var/ossec/logs/alerts/alerts.json
  • Filters alerts by severity level threshold (default: level > 8)
  • Always captures 6-digit custom rules (100000–999999) regardless of level
  • Deduplicates alerts to prevent redundant AI calls
  • Sends each unique alert to Claude AI for security analysis
  • Saves a human-readable .txt report per alert
  • Writes a machine-readable JSON event to claude_analysis.json
  • Filebeat ships that JSON file to the Wazuh Indexer (OpenSearch)
  • Reports appear in the Wazuh Dashboard under the wazuh-claude-* index

4. Installation

Step 1 — Copy the script

sudo cp wazuh_claude_analyzer-V3.py /usr/local/bin/
sudo chmod +x /usr/local/bin/wazuh_claude_analyzer-V3.py

Step 2 — Create output directories

sudo mkdir -p /var/ossec/logs/claude_reports
sudo chmod 755 /var/ossec/logs/claude_reports

Step 3 — Install Python dependency

pip install anthropic

Step 4 — Test run

export ANTHROPIC_API_KEY='sk-ant-...'
sudo -E python3 /usr/local/bin/wazuh_claude_analyzer-V3.py --mode once

Step 5 — Optional: run as a systemd service (watch mode)

Create /etc/systemd/system/wazuh-claude.service:

[Unit]
Description=Wazuh Claude AI Alert Analyzer
After=wazuh-manager.service

[Service]
Type=simple
EnvironmentFile=/etc/environment
ExecStart=/usr/bin/python3 /usr/local/bin/wazuh_claude_analyzer-V3.py --mode watch
Restart=on-failure
RestartSec=10
User=root

[Install]
WantedBy=multi-user.target

Enable and start:

sudo systemctl daemon-reload
sudo systemctl enable wazuh-claude
sudo systemctl start wazuh-claude
sudo systemctl status wazuh-claude

6. How Alert Filtering Works

An alert is selected for analysis if it meets either of these conditions:

Condition A — High severity level

rule.level > ALERT_LEVEL_THRESHOLD  (default: level > 8)

This captures all standard Wazuh built-in rules that are firing at high severity. Wazuh rule levels range from 0 to 15.

Condition B — 6-digit custom rule

100000 <= rule.id <= 999999

Wazuh reserves the 100000–999999 range for organisation-defined custom rules. These are always captured regardless of their assigned level. This ensures that custom detections — which may be assigned a low level during development (e.g., level 3, rule 999900) — are never silently skipped.

Example

Rule ID Level Selected? Reason
5710 10 ✅ Yes Level 10 > threshold 8
19005 9 ✅ Yes Level 9 > threshold 8
999900 3 ✅ Yes 6-digit custom rule
1002 5 ❌ No Level 5 ≤ threshold, not custom
100001 2 ✅ Yes 6-digit custom rule

7. Deduplication Logic

Run-once mode

Before sending any alert to Claude, all matched alerts are deduplicated using a fingerprint:

fingerprint = SHA256(rule_id | agent_id | src_ip | timestamp_minute)[:16]

Alerts sharing the same fingerprint within the file are suppressed. A summary is printed at the end showing how many duplicates were removed.

Watch mode

An in-memory dictionary {fingerprint: epoch_timestamp} persists across poll cycles. Any alert whose fingerprint was already seen within DEDUP_WINDOW_SECONDS (default 300s) is silently skipped. Entries older than the window are purged each poll cycle to keep memory bounded.

The dedup window is tunable at runtime:

sudo -E python3 wazuh_claude_analyzer-V3.py --mode watch --dedup-window 600

8. Running the Analyzer

One-time analysis of existing alerts

export ANTHROPIC_API_KEY='sk-ant-...'
sudo -E python3 wazuh_claude_analyzer-V3.py --mode once

Watch mode (continuous monitoring)

sudo -E python3 wazuh_claude_analyzer-V3.py --mode watch

Watch mode starts at the end of the alerts file and only processes new alerts as they arrive. Existing alerts are not re-processed.

Custom level threshold

# Capture all alerts above level 5 (plus all custom rules)
sudo -E python3 wazuh_claude_analyzer-V3.py --mode once --level 5

Capture rule 999900 (level 3)

Rule 999900 is a 6-digit custom rule so it is always captured automatically without any extra flags:

sudo -E python3 wazuh_claude_analyzer-V3.py --mode once

The console output will show [CUSTOM RULE] tag:

Rule ID    : 999900  🔧 [CUSTOM RULE]
Level      : 3  (threshold >8)

Text reports only (no OpenSearch feed)

sudo -E python3 wazuh_claude_analyzer-V3.py --mode once --no-opensearch

9. Filebeat Integration

Filebeat ships the claude_analysis.json file directly to the Wazuh Indexer (OpenSearch). No Logstash is required.

Setup

1. Create the inputs directory

sudo mkdir -p /etc/filebeat/inputs.d

2. Create the Claude input file

sudo tee /etc/filebeat/inputs.d/claude_analysis.yml << 'EOF'
- type: log
  enabled: true
  paths:
    - /var/ossec/logs/claude_reports/claude_analysis.json
  json.keys_under_root: true
  json.overwrite_keys: true
  json.add_error_key: true
  tags:
    - claude-analysis
  fields_under_root: true
  fields:
    input_type: claude_analysis
EOF

3. Update /etc/filebeat/filebeat.yml

Replace the output.elasticsearch block with one that routes Claude events to their own index:

output.elasticsearch:
  hosts: ["https://127.0.0.1:9200"]
  protocol: "https"
  username: "admin"
  password: "admin"
  ssl.verification_mode: "none"
  ssl.certificate_authorities: ["/etc/filebeat/certs/root-ca.pem"]
  ssl.certificate: "/etc/filebeat/certs/wazuh-server.pem"
  ssl.key: "/etc/filebeat/certs/wazuh-server-key.pem"
  indices:
    - index: "wazuh-claude-%{+yyyy.MM.dd}"
      when.contains:
        tags: "claude-analysis"
    - index: "wazuh-alerts-4.x-%{+yyyy.MM.dd}"

Also ensure the filebeat.config.inputs block is present (add if missing):

filebeat.config.inputs:
  path: /etc/filebeat/inputs.d/*.yml
  reload.enabled: true
  reload.period: 10s

4. Fix file permissions

sudo chmod 644 /var/ossec/logs/claude_reports/claude_analysis.json
sudo chmod 755 /var/ossec/logs/claude_reports/

5. Test and restart

sudo filebeat test config -e    # must return "Config OK"
sudo filebeat test output        # must return "connection OK"
sudo systemctl restart filebeat
sudo systemctl status filebeat

6. Verify the index was created

sleep 15 && curl -sk -u admin:admin \
  https://127.0.0.1:9200/_cat/indices/wazuh-claude-* | column -t

Expected output:

yellow  open  wazuh-claude-2026.03.23  xxx  1  1  2  0  91.8kb  91.8kb

10. Wazuh Dashboard Setup

Create the index pattern

  1. Open the Wazuh Dashboard in your browser
  2. Click ☰ (hamburger menu, top left)
  3. Navigate to Stack Management → Index Patterns
  4. Click Create index pattern
  5. Enter wazuh-claude-* → click Next step
  6. Set Time field to @timestamp
  7. Click Create index pattern

View reports in Discover

  1. Click ☰ → Discover
  2. Select index wazuh-claude-* from the top-left dropdown
  3. Set time range to Last 24 hours (top right)

Useful search filters

Search query What it shows
rule.id : 999900 All reports for rule 999900
rule.level : 3 All level-3 alerts
agent.name : MSS-DC-MISP All alerts from a specific agent
claude.false_positive_likelihood.label : High High FP risk alerts

    "affected_assets": "...",
    "severity_assessment": "...",
    "immediate_actions": ["step1", "step2", "step3"],

False positive score mapping

Label Score Meaning
Low 1 Claude is confident this is a real threat
Medium 2 Some indicators suggest a possible FP
High 3 Likely benign activity, review before acting

12. Troubleshooting

API key not found

Error:

[ERROR] ANTHROPIC_API_KEY environment variable is not set.

Fix:

export ANTHROPIC_API_KEY='sk-ant-...'
sudo -E python3 wazuh_claude_analyzer-V3.py --mode once
# Note: -E preserves environment variables under sudo

No alerts found

Message:

No qualifying alerts found (level >8 or 6-digit custom rule).

Causes and fixes:

# Check how many alerts exist and their levels
sudo cat /var/ossec/logs/alerts/alerts.json | python3 -c \
  "import json,sys; alerts=[json.loads(l) for l in sys.stdin if l.strip()]; \
   print(f'Total: {len(alerts)}'); \
   [print(a['rule']['id'], a['rule']['level']) for a in alerts[:10]]"

# Lower the threshold to capture more alerts
sudo -E python3 wazuh_claude_analyzer-V3.py --mode once --level 3

Filebeat not picking up claude_analysis.json

Symptom: curl shows no wazuh-claude-* index after restart.

Diagnosis:

sudo journalctl -u filebeat --no-pager -n 30 | grep "Configured paths"

Expected (2 inputs):

Configured paths: [/var/ossec/logs/alerts/alerts.json]
Configured paths: [/var/ossec/logs/claude_reports/claude_analysis.json]

If only 1 input is shown:

# Confirm inputs.d file exists
sudo cat /etc/filebeat/inputs.d/claude_analysis.yml

# Confirm filebeat.yml has the config.inputs block
sudo grep -n "inputs.d" /etc/filebeat/filebeat.yml

# Fix permissions
sudo chmod 644 /var/ossec/logs/claude_reports/claude_analysis.json

# Restart
sudo systemctl restart filebeat

Duplicate output.elasticsearch in filebeat.yml

If filebeat.yml has two output.elasticsearch blocks (from earlier edits), Filebeat will fail silently. Keep only one.

sudo grep -n "output.elasticsearch" /etc/filebeat/filebeat.yml
# Should return exactly 1 line

Index exists but no data in Dashboard

# Check document count directly
curl -sk -u admin:admin \
  "https://127.0.0.1:9200/wazuh-claude-*/_count" | python3 -m json.tool

# Check time range in Dashboard — set to "Last 7 days"
# The @timestamp field must be selected as the time field in the index pattern

Rate limit error

[ERROR] Claude API rate limit reached. Retry later.

The script will continue processing the next alert. Rate-limited alerts are saved to the text report with the error message. Re-run in --mode once after a few minutes to retry.
